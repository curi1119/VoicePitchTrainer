# アーキテクチャ

## 全体像

ブラウザ完結の SPA。サーバ・DB・外部 API なし。音声処理はすべてクライアントの Web Audio API で行う。

```
[マイク] → getUserMedia → ローパスフィルタ(1200Hz) → AnalyserNode (fftSize 4096)
                                       │  requestAnimationFrame で約60fpsポーリング
                                       ▼
                         YIN ピッチ検出 (audio/pitch-detector.ts)
                                       ▼
                         多段平滑化 (audio/smoothing.ts)
                         メディアン9 → EMA 0.18 → ジャンプ棄却 → 音名ヒステリシス
                                       ▼
                    ┌──────────────────┴──────────────────┐
              モード判定 (modes/)                     UI 表示 (components/)
              single / scale の状態機械               ピッチグラフ・88鍵
                                       ▲
[スピーカー] ← 音の合成 (audio/synth.ts) ← 出題音・ガイド音・トライアド・効果音
```

## レイヤ構成(設計原則)

```
pitch-trainer/
├─ src/
│  ├─ audio/        # React 非依存の純粋ロジック層(テスト対象)
│  │  ├─ pitch-detector.ts   # YIN 実装(CMNDF + 放物線補間)
│  │  ├─ smoothing.ts        # メディアン + EMA + ジャンプ棄却 + ヒステリシス + 発声ゲート
│  │  ├─ synth.ts            # ビープ/簡易ピアノ合成、トライアド、成功/不正解音
│  │  ├─ sampled-piano.ts    # サンプルピアノ (smplr + SplendidGrandPiano) のロードと再生
│  │  ├─ mic.ts              # getUserMedia + ローパス + AnalyserNode
│  │  ├─ level.ts            # 入力レベル (RMS)
│  │  ├─ output.ts           # AudioContext とマスター音量(全出力の経由点)
│  │  └─ notes.ts            # MIDI ↔ 周波数 ↔ 音名 の変換ユーティリティ
│  ├─ modes/        # 判定ロジック。純粋 TS(テスト対象)
│  │  ├─ single.ts           # 単音発声: キープ/不正解/リトライの状態機械
│  │  └─ scale.ts            # 音階練習: ラウンド進行と音ごとの判定
│  ├─ components/   # PitchGraph(時間×音高グラフ)/ Piano / JudgeBar / 各モードの UI
│  │                 #   pitch-graph-math.ts はグラフの座標・レンジ計算(純関数・テスト対象)
│  ├─ config.ts     # バランス調整値を集約(下表)
│  └─ App.tsx
├─ tests/           # Vitest
└─ vite.config.ts
```

原則:

1. **`audio/` と `modes/` は React・DOM に依存しない**(mic.ts の getUserMedia などブラウザ API は可、React の import は不可)
2. **調整値は config.ts に集約**し、コードにマジックナンバーを散らさない
3. **判定ロジックと調整値はプロトタイプ(`prototype/`)由来の確定値を維持**する。UI は docs/ui_redesign.md 以降プロトタイプから独自進化している(プロトタイプは判定挙動の比較用リファレンス)

## React との接続(レンダリング方針)

毎フレーム(60Hz)更新される値を React state に入れると再レンダリングが洪水になるため、更新頻度でルートを分ける:

- **60Hz 系**(ピッチグラフの Canvas・ホールド進捗バー): rAF ループから ref / Canvas を直接更新し、React の再レンダリングを通さない
- **離散イベント系**(表示音名の切替・判定確定・スコア・モード切替): React state。音名はヒステリシス(±65セント)通過後にしか変わらないので更新頻度は低い
- 数値テキスト(周波数・セント)はプロトタイプ同様 **10Hz に間引き**
- 動的な座標・角度(鍵盤の絶対配置、針の回転)は Tailwind ではなくインライン style で与える
- マイク+検出のメインループはアプリ全体で1本。カスタムフック(例: `useMicPitch`)が React 側の購読窓口

## config.ts に集約する主な調整値(プロトタイプの確定値)

| 定数 | 値 | 意味 |
|---|---|---|
| `YIN_THRESHOLD` | 0.15 | CMNDF(累積平均正規化差分)の採用閾値 |
| `F_MIN` / `F_MAX` | 55 / 1200 Hz | 検出レンジの既定(A1〜ソプラノ上限。レンジ未指定時) |
| 検出音域プリセット | 男性 D2〜D5 (71-605) / 女性 C3〜D6 (127-1210) / 全域 A1〜E6 (54-1330) Hz | ヘッダー「音域」セレクトで選択(`PITCH.DETECT_RANGES`、既定=全域、localStorage `detect-range`)。声域に上限を合わせると高次倍音への誤ロックを構造的に防ぐ。男性上限605は C3 の第5倍音654(≒E5)を外す値(下の調整履歴参照) |
| RMS ゲート | 0.003(可変 0.0012〜0.02) | これ未満の音量は無音として棄却。ヘッダー「感度」スライダーで可変(`RMS_GATE_MIN/MAX`、対数補間、初期値 0.003、localStorage `detect-sensitivity`)。敏感ほど小さい声を拾うがノイズも拾いやすい |
| 救済閾値 | 0.30 | 閾値未達でも CMNDF 全域最小がこれ未満なら採用 |
| 救済の最低 RMS | 0.01 | これ未満の低音量では救済ルールを無効化(誤ラグ防止。調整履歴参照) |
| `fftSize` | 4096 | 約85ms窓 @48kHz(男性低音の分解能確保) |
| ローパス | 1200 Hz | マイク入力の高次倍音抑制 |
| メディアン窓 | 9 フレーム | 約150ms の外れ値除去 |
| EMA 係数 | 0.18 | 時定数 約90ms |
| ジャンプ棄却 | 0.7 半音 | 直近3フレーム安定なら意図的な移動として即追従 |
| 音名ヒステリシス | 0.65 半音 | 表示音名の切替閾値(境界チラつき防止) |
| 発声開始 / 終了ゲート | 4 / 8 フレーム | 約65ms / 約130ms |
| オクターブ連続性 | ±0.7半音 / 30フレーム | 確立ピッチのちょうど整数オクターブ(±12/±24半音)に当たる検出を倍音/サブ倍音の誤りとみなし折り返す。連続30フレーム(約490ms)で意図的移動として受理(`SMOOTHING.OCTAVE_FOLD_TOLERANCE/CONFIRM_FRAMES`、2026-06-16)。第5倍音等の非オクターブロックは音域プリセット担当 |
| テキスト更新 | 10 Hz | 数値のチラつき防止 |
| グラフ表示窓 | 6 秒 | ピッチグラフの横軸(右端が現在) |
| グラフレンジ追従 | lerp 0.08, 最小24半音(2オクターブ), 余白3半音 | 縦軸の自動パン(`GRAPH.*`)。最小スパンは 14→24(2026-06-13、ズームインしすぎて落ち着かないため) |
| `QUIZ_TONE_DUR` | 3.0 秒 | 出題音の長さ(再生終了 +0.2 秒後に判定開始) |
| `OK_HOLD_MS` | 1500 | 正解: ±50セント以内をキープする時間 |
| `NG_HOLD_MS` | 1200 | 不正解: 外れた音程のまま安定する時間(キープ外れは2倍速で減少) |
| `AUTO_QUIZ_DELAY_MS` | 2000 | 自動出題(オプション)で判定確定から次の出題までの待ち |
| 音階判定 | ±60セント, `max(6, voiced×0.35)` | 1音ごとの合格条件 |
| 音階ガイド音長 | `max(0.5, 拍×0.95)` 秒 | 短いとピチカート化する不具合対策 |
| トライアド | 2拍再生 + 半拍空け | ラウンド開始前に調を提示 |

| `SAMPLED_VELOCITY` | 108 | サンプルピアノのベロシティ 0-127(音色の明るさにも影響。90→108) |
| `SAMPLED_BOOST` | 1.6 | 追加ブースト倍率。コンプレッサー経由で音割れを防止(2026-06-13 スマホ音量対策) |

## 調整履歴(なぜこの値か)

> 企画フェーズの引継書 HANDOVER.md は内容を本書へ移管のうえ 2026-06-13 に削除した(git 履歴で参照可能)。

| 項目 | 経緯 |
|---|---|
| 正解キープ時間 | 0.5秒 →(短すぎ)→ 2.0秒 →(長すぎ)→ **1.5秒で確定**(プロトタイプ期) |
| 不正解確定時間 | 0.8秒 → 1.5秒 → **1.2秒**(キープ時間に比例調整) |
| 出題音の長さ | 1秒 → **3秒**(開発者要望) |
| メーター感度 | 初版は敏感すぎ → 多段平滑化(メディアン+EMA+ヒステリシス)で「ちょうどよい」評価 |
| ガイド音長 | 最大0.35秒 → **1拍の95%**(短いと合成ピアノがピチカート化する不具合の解消) |
| ピッチ検出方式 | 自己相関(ACF2+)→ **YIN**(男性の低い声で第5倍音にロックし E5 等と誤検出する障害の解消) |
| ピアノ音色 | 倍音加算合成 → **サンプル音源 smplr + SplendidGrandPiano**(2026-06-13。音質不満の解消) |
| サンプル音量 | volume 100 / velocity 90 → **127 / 108 + 1.6倍ブースト**(2026-06-13。スマホで音量不足) |
| メイン表示 | アナログ針メーター+波形 → **ピッチグラフ(時間×音高)**(2026-06-13。docs/ui_redesign.md) |
| 入力感度 | RMS ゲート 0.015 → 0.006 → **0.003**(2026-06-13。小さい声を拾えない感度不足の解消、実機評価で段階的に引き下げ。周期性の品質判定は YIN_THRESHOLD 0.15 が担っており判定精度は維持) |
| 過大入力時の倍音ロック対策(検出音域設定) | 大声を近距離で入れると波形が歪んで高次倍音が支配的になり、YIN がそれにロックして「男性低音 C3〜F3 が E5/F5 付近=第5倍音に化ける」誤判定が発生(2026-06-13、PC マイクで再現。波形キャプチャでは F5≈681Hz に張り付き、声量を下げる/マイクから離すと解消)。歪んだ信号は CMNDF 上で本物の高音と区別不能で検出器側では直せない。当初 LPF 前の生信号でクリップを検出し棄却する案を試したが、ハードクリップ(レール張り付き)に至らないソフトな歪みでは発火せず無効だった。代わりに**ユーザが声域を選べる「音域」設定**を追加し、検出レンジの上限を下げて高次倍音帯を探索対象から外す方式に(男性=上限605Hz≒D5上で、C3 の第5倍音654=E5 を探索範囲外にする)。プリセットは男性 D2〜D5 / 女性 C3〜D6 / 全域 A1〜E6 の3種。回帰テストで「各プリセットでレンジ内は誤差±2セント以内」「倍音突出信号で上限超え検出が起きない」を固定。**根本的にマイク入力が歪む場合は OS のマイク入力レベルを下げる/ブーストを切るのが確実** |
| マイク感度のユーザ調整 | 環境差(マイクの拾い具合)を吸収するため、`RMS_GATE` をヘッダー「感度」スライダーで可変に(2026-06-13)。音域(=周波数範囲)とは独立した軸で、YIN は振幅非依存のため感度変更は音程精度に影響しない。0.0012〜0.02 を対数補間し初期値は従来の 0.003。敏感側はノイズ誤検出とのトレードオフがあるため範囲を絞っている |
| 低音量時の救済無効化 | RMS < 0.01 では CMNDF 救済(0.30)を適用しない(2026-06-13。感度向上後「長い発声の末尾でピッチがずれる/拾わなくなる」障害が発生。減衰発声のシミュレーションで再現し、ノイズ優勢フレームを救済ルールが誤ラグで拾っていたと特定。修正により平滑化後±60c超ずれ 54→0 フレーム) |
| 単音発声 UI の整理 | スコア表示「正解 N/M」と「答えを見てパス」ボタンを削除(2026-06-16。前者は有益でない、後者は「出題」と役割が重複)。**判定ロジック(`single.ts`)は不変で `pass()`・`ok/all` カウントはテスト用に残置**、UI 側の表示/操作のみ撤去。あわせて2オプションを追加: **チューナーを隠す**(出題〜判定確定まで音高メーター=PitchGraph キャンバスのみをオーバーレイで覆い、耳だけで合わせる練習。下の JudgeBar=メッセージ/キープバーは隠さない。`tunerCovered` で制御)、**自動出題**(判定確定の `AUTO_QUIZ_DELAY_MS`=2秒後に次を自動出題。メインループの stale closure 回避に `handleQuizRef`/`autoQuizRef` を使用)。両オプションは localStorage(`single-hide-tuner` / `single-auto-quiz`)に保存。なお「もう一度」(`replay()`)は再生中マイクがアプリ音を拾うため判定を止める設計で、キープバーが固まって見えないよう hold/wrong をリセットして仕切り直しにする |
| 音階ラウンド数・折り返し / UI 改善 | 音階練習に**ラウンド数**(上昇周回数、既定10)と**折り返し**(往復・無限ループ)を追加(2026-06-16)。折り返しは `ScaleMode` に進行方向 `dir`(±1)と開始音 `startBase` を持たせ、上端で `dir`反転・下端(開始音)で再反転する方式。停止条件は折り返しOFF時のラウンド数到達のみで、最高音 MIDI 96(`MAX_TOP_MIDI`)は常に安全弁。あわせてスマホUI改善: ヘッダー「音域」セレクトの音域表記(D2〜D5等)を `?` ポップオーバーへ逃がし幅を縮小+コントロール群を `flex-wrap` 化してマイク開始ボタンのはみ出しを解消、鍵盤モードに CSS オーバーレイの全画面表示を追加。設定の localStorage 保存も拡充(下記「既知の制限」参照) |
| 倍音↔基音の往復対策(オクターブ連続性) | 発声中に基音と倍音(主に1オクターブ上)を行き来する/女声でオクターブ下に落ちる誤検出(2026-06-16)。**当初は検出器(YIN)側の単フレーム補正を試みたが撤回**: 「採用ラグの2倍に深いディップがあれば基音とみなして降ろす」案は、(1)周期信号は2倍・3倍周期にも必ず深いディップが出るため正解を必ずオクターブ下へ誤爆する、(2)実声では正しいピックの `cm[tau]`・スペクトル下位倍音比が倍音ロックと重なり**単フレームでは分離不能**、と合成信号で実証された(きれいな合成音では分離できても実声で破綻し、女声がオクターブ下に落ちる回帰を生んだ)。これは pYIN が YIN に HMM 平滑化を足す理由そのもの。代わりに**平滑化(`smoothing.ts`)側で時間方向の文脈を使う**方式に。確立ピッチのちょうど整数オクターブ(±12/±24半音、許容±0.7)に当たる検出を倍音/サブ倍音の誤りとみなし確立オクターブへ折り返し、連続して続けば意図的なオクターブ移動として受理する。ジッター付き合成系列で「往復・単発スパイク・サブ倍音落ち」を 0 誤り化、素直な維持の精度劣化なし、意図的移動は追従を確認。**第5倍音ロック(歪み由来=基音が入力から消失、非オクターブ)は対象外で、音域プリセット(男性=上限605Hzで E5 を除外)とマイク入力レベル低減が担当**。受理までの確認フレーム数は当初6(約100ms)だったが、iPhone 実機テストで 200〜450ms 持続するオクターブ上スパイク(矩形状に C4→C5 へ跳ね保持して戻る倍音ロック)が残ったため **30(約490ms)へ引き上げ**(2026-06-16)。発声の途切れ後の再発声は ema クリアで即新オクターブを確立するため、この確認が効くのは「連続発声中の不意のオクターブ跳び≒倍音ロック」のみで、意図的移動(通常ブレスを伴う)への副作用は小さい |

## テスト戦略

- **YIN 回帰テスト(必須)**: 人声を模した合成信号(基音が弱く倍音が強い: 振幅 [0.05, 0.15, 0.30, 0.28, 0.25, 0.18]、位相 h×0.7、ノイズ ±0.005)で C2〜C5 を検出できること。プロトタイプ期の実測誤差は 0.1 セント以内、処理時間 約2.3ms/フレーム
- smoothing: ジャンプ棄却・ヒステリシス・発声ゲートの境界値テスト・オクターブ連続性(往復の折り返し / 持続的オクターブ移動の受理 / 非オクターブ移動の素通し)
- modes: 判定の状態遷移(キープ→正解 / 外れ安定→不正解 / リトライ→正しい音でスコア据え置き)を時系列入力でテスト
- コンポーネントの自動テストは当面なし(目視 + 実機確認)

## 既知の制限・将来課題

- ピアノ音源はサンプル化済み(2026-06-12、smplr + SplendidGrandPiano)。サンプルは公式 CDN(smpldsnds)から取得し、Cache API で永続キャッシュするため一度ロードすればオフラインでも鳴る。**Capacitor で同梱する場合は smplr の `baseUrl` オプションで切り替える**。音質に不満が出たら Salamander Grand Piano(CC BY 3.0・要クレジット表記)への乗り換えが次候補。アセットはライセンス的に問題ないもののみ利用可
- 音階練習でガイド音 ON + スピーカー使用時は自分のマイクがアプリ音を拾いうる(ヘッドホン推奨で運用)
- AudioWorklet 化(検出処理のオーディオスレッド移行)は見送り中。現状メインスレッドで約 2.3ms/フレームと実用十分
- 設定の localStorage 永続化(2026-06-16 拡充): 音量(`master-volume`)・音域(`detect-range`)・感度(`detect-sensitivity`)に加え、単音発声(`single-preset` / `single-low` / `single-high` / `single-hide-tuner` / `single-auto-quiz`)と音階練習(`scale-pattern` / `scale-base` / `scale-bpm` / `scale-guide` / `scale-round-count` / `scale-turnaround`)を保存。**未保存: 音色・スコア**
- 未着手の将来項目: 音名表記の切替(ドレミ等)、練習/テストモード切替(不正解即確定 1.2 秒は初心者に厳しい可能性)、Capacitor によるモバイルアプリ化
